# Deadmouse 仓库级工作方式

本文件定义进入本仓库后应默认遵守的长期规则。

它不是人格提示词，也不是一次性任务说明，而是本仓库的开发方式、边界和收口标准。

## 1. 真相源

- 已确认通过的 `spec` 是任务真相源。
- `spec/用户审阅/宪法原则/` 是仓库级长期原则的上游来源。
- 当文档、测试、代码冲突时，先检查 `spec` 与宪法原则，再让三者重新一致。
- 不接受“代码先改，文档以后再补”。

## 2. 默认主路径

本仓库默认按下面顺序推进：

`spec -> test -> implementation -> verification -> closeout`

具体要求：

- 先完整理解目标、范围、约束、验收标准、失败标准、边界条件和范围外事项，再开始实现。
- 先抓主路径、主约束、主风险、主验收，再处理枝节。
- 优先把“会不会被改坏”写成失败测试、检查脚本或等价验证，再写实现。
- 没有真实验证通过，不能判定为完成。

## 3. 设计原则

- 一个文件只做一件事。
- 简单优先，边界优先，解耦优先。
- 简单不是保守，简单是可靠性、可维护性和长期演进的前提。
- 不要为了拆而拆，也不要把本应独立的职责硬塞回一个大文件。
- 主循环只保留调度职责，不吞模块细节。
- 新功能优先长在明确模块、目录或正式扩展口，不优先堆进已有大文件。

参考来源：

- `ref/prd/一个文件只做一件事.md`
- `spec/用户审阅/宪法原则/18-主循环和文件都不能长胖.md`

## 4. 扩展原则

- 新扩展优先走正式扩展口，不优先靠继续堆 prompt 文案。
- 工具能力走 `src/tools/`
- 技能能力走 `src/skills/`
- MCP 能力走 `src/mcp/`
- 宿主注入与运行时边界走 `src/host/`
- 不允许宿主逻辑绕进核心内部模块形成散装胶水。

参考来源：

- `spec/用户审阅/宪法原则/17-扩展靠事件生长.md`

## 5. Prompt 相关规则

- system prompt 是正式结构，不是散装长文。
- 静态层与动态层的正式边界在：
  - `src/agent/prompt/`
  - `src/agent/promptSections.ts`
- 人格、架构思维、执行契约、运行时状态应按块组织，不要揉成一坨。
- 修改 prompt 结构、静态块、动态块或 contract 文案时，必须同步检查并更新对应测试。

最低要求：

- `tests/system-prompt-contract.test.ts`

必要时继续检查：

- `tests/runtime-lightweight-context.test.ts`
- 其他直接依赖 prompt 文案或 prompt 指标的测试

## 6. 验证与收口

- “写了文件”不等于“做成了”。
- 收口必须依赖真实产物、真实命令、真实状态、真实验证，不依赖模型自述。
- 只要关键验证没过、关键文件缺失、关键行为不可用、关键结果不可读，就不能 finalize。
- 如果任务显式给出了 acceptance / closeout 契约，必须按契约继续检查。

参考来源：

- `spec/用户审阅/宪法原则/19-先写失败测试再写实现.md`
- `spec/用户审阅/宪法原则/21-没验过就不能收口.md`

## 7. 实施偏好

- 优先复用现有实现、现有模块、现有工具和成熟方案。
- 小步推进，分块修改，逐块验证。
- 不把大量不相关改动糊成一个大 patch。
- 不把计划当结果，不把解释当完成，不把猜测当事实。
- 任何阶段都要维护文档、实现、测试的一致性。

## 8. 与项目 Owner 的沟通方式

下面是沟通方式示例，目标是用项目 Owner 能理解的运行例子讲清楚问题，不要先讲代码。

可以，我用“运行时会发生什么”的方式讲，不讲代码。

**1. 技能加载太硬**
现状例子：用户说“帮我看这个 docx 文档并总结问题”，模型想先用普通文件工具看目录、确认文件是否存在，但机器判断这是 docx-review 类任务后，可能先拦住它，要求必须加载某个技能。  
理想状态：机器提醒“这个任务可能适合加载 docx 技能”，但模型仍然可以先查文件、看路径、确认输入，再决定要不要加载技能。

**2. 任务调度太像导演**
现状例子：用户给了一个稍长的任务，比如“研究项目、找问题、给方案、必要时改代码”，机器可能因为任务长、结构复杂，就自动倾向分给子代理或队友。  
理想状态：机器只告诉 Lead：“这个任务可能适合拆分或并行”，但是否真的拆、拆给谁、什么时候拆，由 Lead 模型判断。

**3. 正则意图识别太重**
现状例子：用户写“spawn 一个 worker 看看测试”，机器容易把这句话识别成明确委派请求；但用户可能只是随口描述，模型本来可以先自己判断要不要派。  
理想状态：机器把它标记为“可能有委派意图”，而不是直接改变执行路线。

**4. 计划门槛偏硬**
现状例子：模型只是想改一个错别字、修一个小配置，直接调用编辑工具时，机器可能说“先写 todo”。  
理想状态：大改动、多个文件、危险命令前要求计划；小修小补可以先做，机器只提醒“建议补一个简短计划”。

**5. 工具排序有隐性偏向**
现状例子：用户说“查一下某网页内容”，工具虽然都可见，但浏览器工具被排到很前，模型容易以为必须先开浏览器。  
理想状态：机器可以把 HTTP、下载、浏览器都摆出来，并提示“轻量抓取优先，交互页面再用浏览器”，但不塑造唯一正确路线。

**6. 收口方向是对的**
现状例子：模型说“我已经完成了”，但文件没改、测试没跑、证据没记录，机器会阻止它直接完成。  
这个应该保留，因为这里不是限制模型路线，而是在防止“嘴上完成”。

**7. 循环提醒方向是对的**
现状例子：模型连续三次跑同一个失败命令，输出差不多，机器会提醒它重复了。  
理想状态不是简单说“禁止再跑”，而是说“你已经重复失败，请先总结失败原因，然后换一种办法继续”。

**8. 后台任务建议要统一**
现状例子：模型运行一个可能较久的测试命令，机器以前可能强迫它改用后台。  
理想状态：机器提示“这个可能较久，后台更合适”，但如果模型判断前台跑更清楚，也应该允许。

**9. 多代理边界不够收**
现状例子：Lead 派了一个子代理，后台又在跑任务，队友也有结果，但主任务还没合流，某个执行通道可能已经觉得自己完成了。  
理想状态：任何子代理、队友、后台都只能交结果；最终是否完成，只能 Lead 根据证据统一判断。

**10. 总体运行效果**
现在有些地方像“机器看到关键词后替模型拐弯”。  
我希望改成“机器把路牌、仪表盘、刹车做好，但方向盘始终在模型手里”。

## 9. 本文件的定位

- 本文件只写仓库级长期有效规则。
- 一次性任务要求放到任务 prompt 或对应 spec。
- 具体模块细节放到 `spec/技术实现/` 或模块内局部文档。
- 人格、语气和单次任务口号不写进本文件。
