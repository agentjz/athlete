# Kitty 基础工具循环测试计划

## 目标

- [ ] 验证 `agent` 模式是否真正进入 Pi 风格四件套：`read / edit / write / bash`。
- [ ] 验证体验是否接近成熟工程师：快速定位、局部阅读、直接修改、立刻对账、马上验证。
- [ ] 验证生态能力不会在普通编码任务中抢戏。
- [ ] 验证工具失败时只给必要事实，不输出大段协议噪音。
- [ ] 通过多轮真实任务暴露问题，发现旧残留、分散治理、特判、硬编码就直接删除或重写。

## 固定原则

- [ ] 使用构建产物运行：`node dist/cli.js`。
- [ ] 不把 `live:ecology` 固定 harness 当主循环。
- [ ] 不要求模型输出固定报告。
- [ ] 每轮提示词都变化，模拟真实使用。
- [ ] 成功路径追求短、快、直接。
- [ ] 失败路径只展开必要证据。
- [ ] 机器层只记录事实，不替模型做决策。
- [ ] 发现旧设计残留时，先物理删除，再按当前设计重写。

## 每轮流程

- [ ] 先构建，确保测试对象是最新 `dist`。
- [ ] 用一个自由编码任务运行 `node dist/cli.js`。
- [ ] 观察终端日志、工具调用顺序、模型可见输出、失败提示和最终变更。
- [ ] 判断是否符合：`bash 定位 -> read 精读 -> edit/write 修改 -> bash diff/test`。
- [ ] 如果出现绕路、噪音、旧工具、生态泄漏、特殊硬编码、保守门槛，直接改代码。
- [ ] 改完后重新构建。
- [ ] 跑必要自动测试。
- [ ] 再用新的自由任务进入下一轮。

## 第一轮任务

- [ ] 提示词：让 Agent 在一个小范围代码问题中自行定位相关文件，修改一处行为，并运行最小验证。
- [ ] 重点观察：是否先用 `bash` 快速定位，而不是乱读大量文件。
- [ ] 重点观察：是否只读必要片段。
- [ ] 重点观察：是否使用 `edit` 或 `write`，不出现旧工具名。
- [ ] 重点观察：改完是否用 `bash` 看 diff 或运行测试。
- [ ] 重点观察：工具输出是否短，不把机器证据堆给模型。

## 第二轮任务

- [ ] 提示词：让 Agent 新建一个很小的测试或文档片段，并验证仓库状态。
- [ ] 重点观察：新建是否走 `write`。
- [ ] 重点观察：搜索、目录、Git 都是否交给 `bash`。
- [ ] 重点观察：失败时是否直接给原因，不进入反复解释。

## 第三轮任务

- [ ] 提示词：让 Agent 处理一个需要搜索、多文件阅读、小改和测试的组合任务。
- [ ] 重点观察：复杂一点时是否仍保持四件套节奏。
- [ ] 重点观察：是否出现生态能力误入默认 Coding。
- [ ] 重点观察：是否出现旧兼容路径、旧命名或旧测试暗门。

## 记录格式

- [ ] 每轮只记录短事实：提示词、实际工具顺序、发现的问题、改了什么、验证结果。
- [ ] 不写长报告。
- [ ] 不为了记录而约束 Agent 输出格式。
- [ ] 记录服务于下一轮改代码，不服务于形式化验收。

## 停止标准

- [ ] 连续多轮普通编码任务都稳定走四件套。
- [ ] Agent 默认工具面没有旧工具和生态泄漏。
- [ ] 成功输出足够短。
- [ ] 失败提示足够清楚。
- [ ] 变更后测试通过。
- [ ] 残留扫描没有旧基础工具名、旧 route、旧 prompt、旧兼容概念。
