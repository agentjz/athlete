# 状态与真相源

## 目标

Athlete 要跑长任务、多 Agent 协作和恢复任务，控制面不能继续靠多份 JSON 各自记事。

从本轮开始：

- `task / team / protocol request / background / worktree` 的正式真相源统一进入 SQLite 本地账本
- JSON 不能再参与机器裁决
- JSONL 只保留为审计流、事件流、导出物

## 正式真相源

### SQLite 本地账本

账本文件固定放在：

- `.athlete/control-plane.sqlite`

这是当前唯一正式控制面账本。

控制面判断必须从这里读，不能绕回旧 JSON。

### 本轮进入账本的实体

1. `tasks`
2. `task_dependencies`
3. `team_config`
4. `team_members`
5. `coordination_policy`
6. `protocol_requests`
7. `background_jobs`
8. `worktrees`
9. `ledger_meta`

### 各实体负责回答的问题

#### tasks + task_dependencies

回答：

- 有哪些任务
- 哪些任务被阻塞
- 谁被指派
- 谁正在做
- 任务当前绑定了哪个 worktree
- orchestrator metadata 记录了什么

#### team_config + team_members + coordination_policy

回答：

- 当前团队名是什么
- 有哪些 teammate
- 各自角色、状态、pid、sessionId 是什么
- 当前允不允许 plan decision / shutdown request

#### protocol_requests

回答：

- 哪些正式协作仍在 `pending`
- 哪些已经 `approved / rejected`
- 是谁在什么时间做出的裁决

#### background_jobs

回答：

- 哪些后台命令在跑
- 谁发起的
- 是否 `completed / failed / timed_out`
- pid / exitCode / output / finishedAt 是什么

#### worktrees

回答：

- 哪些 worktree 仍然有效
- 它们在哪个路径、哪个分支
- 状态是 `active / kept / removed`
- 它们和任务的绑定关系是什么

## 审计流与事件流

下面这些 JSONL 继续允许存在，但它们不再是正式真相源：

- `.athlete/team/messages.jsonl`
- `.athlete/team/inbox/*.jsonl`
- `.athlete/worktrees/events.jsonl`

它们的职责是：

- 审计
- 事件回放
- 调试
- 用户可读导出

它们不能决定：

- 任务归属
- 协议状态
- teammate 状态
- background job 状态
- worktree 绑定

## 直接废弃或清理的旧路径

以下旧控制面 JSON/目录从本轮起退出正式状态裁决，并在初始化或清理时允许直接删除：

- `.athlete/tasks/`
- `.athlete/team/config.json`
- `.athlete/team/policy.json`
- `.athlete/team/requests/`
- `.athlete/team/background/`
- `.athlete/worktrees/index.json`

说明：

- 这些路径不是迁移期双写目标
- 这些路径不是回退通道
- 这些路径不是裁决兜底

## 为什么本轮不做长期兼容

Athlete 还是新项目。

这轮的目标不是“平滑一点”，而是把控制面从平行文件堆，收口到正式账本。

如果继续保留：

- 旧 JSON 读取
- 新旧双写
- 平行真相源
- 影子回退路径

就会继续让多 Agent 状态裁决依赖“哪个文件先被读到”，而不是依赖一个正式账本。

这违反：

- `P22-阶段推进必须有机器状态`
- `P24-错误兼容不能高于正确性`
- `P25-新项目不为旧残余保活`

## 并发写入时由谁裁决

由 SQLite 账本裁决，不由工具层裁决，不由 prompt 裁决。

具体规则：

1. 账本层负责事务、唯一约束、条件更新。
2. 任务 claim、worktree 绑定、协议 resolve 这类会冲突的写入，必须落在同一账本事务语义里。
3. tools 只能调用 store；不能自己发明“谁先抢到”的规则。
4. orchestrator 只能读取账本派生结果；不能绕过账本偷看旧文件。

## 为什么这轮是多 Agent 地基，不是单纯换存储

多 Agent 不是“多开几个进程”就结束。

真正难的是以下控制面问题：

- 同一任务不能被两个人同时成功 claim
- 协议请求必须有稳定的 pending / approved / rejected 状态
- worktree 绑定不能因重载或并发写入漂移
- background job 状态不能在恢复时丢失
- orchestrator 必须看到统一真相，而不是多个文件残影

SQLite 账本解决的是这些裁决问题。

所以本轮的本质是：

- 给多 Agent 协作建立正式账本
- 让控制面有单一真相源
- 让重载、恢复、并发冲突都由机器状态收口

不是简单把 JSON 换个格式。

## 本轮明确不动的状态

以下状态不在本轮迁移范围内，继续保留现有机制：

- session
- checkpoint
- verification state
- runtime stats
- tool result artifacts
- channel 专属状态（Telegram / Weixin）

原因：

- 它们不是本轮控制面裁决的核心对象
- 本轮要先把 task/team/protocol/background/worktree 收口
