# 总体架构

## 一句话

Athlete 是一个终端里的智能体运行系统：

- 一个主循环驱动模型与工具
- 一个 SQLite 控制面账本记录正式机器状态
- 一组工具与 worker 构成执行面
- 一层 orchestrator 负责 lead 的任务拆分、派工、等待与合流

## 当前主要分层

### 1. Runtime Kernel

目录：`src/agent/`

负责：

- turn loop
- system prompt 组装
- continuation / compact / recovery
- todo / verification / runtime state
- 最终收口
- 轻量验证信号与 auto-readback

### 2. Tool Plane

目录：`src/tools/` `src/mcp/`

负责：

- 向模型暴露动作
- 参数解析
- 机器约束检查
- 调用底层能力
- 统一治理 metadata、fail-closed 暴露过滤、routing hint 与 signal 校验

### 3. Skill Plane

目录：`src/skills/`

负责：

- skill schema
- discovery
- matching
- loading
- workflow gating

### 4. Control Plane Ledger

目录：`src/control/ledger/` 与 `src/tasks/` `src/team/` `src/background/` `src/worktrees/`

负责：

- task 真相源
- team 真相源
- protocol request 真相源
- coordination policy 真相源
- background job 真相源
- worktree 真相源

正式落盘介质：

- `.athlete/control-plane.sqlite`

### 5. Audit And Event Streams

负责：

- inbox 投递
- 协作消息审计
- worktree 生命周期事件

允许保留为 JSONL 的路径：

- `.athlete/team/messages.jsonl`
- `.athlete/team/inbox/*.jsonl`
- `.athlete/worktrees/events.jsonl`

这些流不是正式裁决真相源。

### 6. Execution Plane

负责真正干活：

- 文件读写
- shell 命令
- 背景任务进程
- teammate worker
- worktree 目录
- MCP 浏览器

### 7. Orchestrator Layer

目录：`src/orchestrator/`

负责：

- 复杂度判断
- 最小任务图落盘
- 从统一账本派生任务 lifecycle 与 wait condition
- self / subagent / teammate / background 路由
- 子任务完成后的 merge / join 推进
- lead slice 开始前的任务组织

## 多 Agent 调度模型

这一层从第 6 轮开始，不再只是“派一点工”，而是正式调度器。

调度器只做四类动作：

1. `split`
   - 把复杂 objective 落成最小可恢复任务图
   - 图仍然落在正式 task ledger 上，不新造平行 orchestrator 状态文件
2. `dispatch`
   - 把 ready task 路由给 `lead / subagent / teammate / background`
   - 路由规则集中在 orchestrator 与控制面派生逻辑里
3. `wait`
   - 当没有 lead-ready work、但已有 teammate / background child 在跑时，明确进入等待
   - 等待条件来自机器派生，不靠 prompt 记忆
4. `merge`
   - 子任务完成后，把父任务推进到下一个 lead step 或 join step
   - 不允许“子任务做完了，但父任务没有正式合流点”

## 调度图的基本约束

1. 一个 objective 对应一组持久化 task graph 节点。
2. task graph 的正式边仍落在 `tasks + task_dependencies`。
3. 每个 orchestrated task 还会带一份持久化 scheduler metadata，用来描述：
   - objective key
   - task kind
   - executor kind
   - background command / delegated teammate / 其他必要调度事实
4. `ready / active / wait / merge readiness / legality` 是从账本派生的机器语义，不是 prompt 文案。
5. subagent 是一次性、同步返回的 child executor；teammate 和 background 是可等待、可恢复的 child executor。
6. worktree 是 execution isolation，不是 task graph 本身；但 teammate lane 一旦进入并行改动，就必须受 worktree 事实约束。

## 当前最重要的架构边界

### 控制面账本回答

- 现在有哪些任务
- 谁该做、谁在做
- 哪些正式协作在 pending
- 哪些后台任务在跑
- 哪些 worktree 仍然有效
- 当前 plan / shutdown 是否允许

### 执行面回答

- 文件怎么改
- 命令怎么跑
- 浏览器怎么导航
- 某个 worker 具体在执行什么

## 关键规则

1. 控制面判断必须从 SQLite 账本读取，不从旧 JSON 读取。
2. 执行面不发明控制面真相。
3. orchestrator 负责调度，不负责具体文件改写。
4. tools 提供动作，skills 提供 workflow，二者不互相替代。
5. 主循环只做调度；closeout、verification signal、workflow gating 这类长期规则要下沉到独立模块。
6. task readiness、ownership、handoff legality、worktree binding reality 由机器从统一账本派生，不靠 prompt 记忆。
7. JSONL 只做审计和事件流，不能回到正式裁决链路。
8. 旧 JSON 真相源如果阻碍主干，可以直接清理，不做长期兼容层。
9. 旧 prompt 型协调只能保留为人类可读提示，不能继续承担 split / dispatch / wait / merge 裁决。
