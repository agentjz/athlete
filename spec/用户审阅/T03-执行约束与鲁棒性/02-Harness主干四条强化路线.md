# Harness主干四条强化路线

## 这份文档回答什么问题

这份文档说明 Deadmouse 的机器层主干已经补齐了哪四类能力，以及这些能力现在是怎样落地的。

## 当前目标与状态

当前项目已经把下面四条主干能力收口为正式能力：

1. 更稳定的文件编辑定位
2. 更完整的写后机器反馈
3. 更可靠的压缩、恢复和续跑
4. 更完整的工具批次执行制度

这份文档描述的是已经完成的结果，不是后续计划。

## 适用范围

这四条主干直接作用于：

- `read_file / edit_file / apply_patch / write_file`
- 共享 `runTurn` 主路径
- 会话现场中的 checkpoint、session diff 和恢复状态
- 一条 assistant 消息内的多工具执行

## 范围外事项

本专项明确不处理：

- UI、TUI、Web 包装
- 审批式权限系统
- 宿主产品面
- provider 产品层包装

## 核心场景

### 1. 小范围精修

当前系统要求先读文件，再基于正式 identity 和锚点编辑，避免在相似文本中改错位置。

### 2. 写后自检

当前写工具会返回 `changedPaths`、diff、diagnostics 和 session diff 摘要，结果不再只靠模型自述。

### 3. 长任务续跑

当前系统在压缩、恢复和继续执行之间保留正式状态；压缩后出现连续空响应时，会进入恢复路径而不是假装成功。

### 4. 一轮多工具执行

当前多工具执行已经按 `prepare -> execute -> finalize` 运行，并且具备 `pendingToolCalls`、并行/串行策略和稳定的结果收口顺序。

`checkpoint.flow` 里增加了正式 run-state 归属（busy/idle），并和 `pendingToolCalls` 同步变化。

## 验收标准

当前专项按下面的结果验收：

- 编辑定位具备正式 identity 与锚点
- 写后反馈进入正式机器结果
- 压缩和恢复后能继续沿正式状态推进
- 多工具批次执行有统一协议和可观察状态
- busy/idle 执行状态可在正式会话现场中读取
- 文档、测试、实现保持一致

## 当前定稿

这四条主干能力已经成为 Deadmouse 当前机器层的一部分。它们服务的是“让系统更稳、更耐跑、更可验证”，而不是继续发散额外产品形态。
