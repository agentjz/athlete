# GPT-5.4 接入与 Provider 适配

## 这份文档回答什么问题

这份文档说明 Deadmouse 当前怎样接模型，以及 GPT-5.4 是否已经按正式路径接入。

## 当前目标与状态

当前项目已经完成两件关键工作：

- 把模型接入边界整理成“通用请求协议层 + Provider 适配层”
- 让 GPT-5.4 按这条正式边界真实接入，而不是挂在特判逻辑上

当前结论已经明确：

- GPT-5.4 已正式接入
- `doctor`、真实 API 请求、真实 Agent 回合都已验证
- 运行配置以项目自己的 `.deadmouse/.env` 和配置系统为准

## 适用范围

这次收口覆盖的是：

- Lead、teammate、subagent 共用的模型请求链路
- `doctor` 的连通性探测
- 运行时的 provider 选择、协议选择和基础请求策略

## 范围外事项

这次收口明确不处理：

- 宿主产品面改造
- 与模型接入无关的功能扩张
- 为provider 残余行为保留长期兼容层
- 把所有 provider 的差异强行抹平

## 核心场景

当前已落地的主场景是：

1. 配置系统读取 `.deadmouse/.env` 中的 provider、model、base URL、API key、thinking、reasoning effort 和 max output tokens。
2. `doctor` 用同一套正式配置完成探测。
3. 运行时按 provider 能力选择正确协议，而不是在核心主路径里写死分支。
4. GPT-5.4 通过当前可用的 `responses` 协议完成真实请求。
5. Deadmouse 基于同一接入链路完成真实 Agent 回合。

DeepSeek 官方 V4 的运行语义是：模型名只表示 `deepseek-v4-flash / deepseek-v4-pro`，普通模式或思考模式由 `thinking=disabled/enabled` 决定；只有思考模式才发送 DeepSeek 官方的 `reasoning_effort=high|max`。如果把 GPT 的 `xhigh` 之类配置拿来给 DeepSeek 用，运行时应该直接报配置错误，而不是偷偷改成另一个值。

模型输出上限由 `.deadmouse/.env` 的统一配置进入请求。DeepSeek 官方 V4 走 `chat.completions` 时映射为 `max_tokens`，GPT-5.4 走 `responses` 时映射为 `max_output_tokens`。

如果 DeepSeek 官方 V4 在思考模式下发起工具调用，运行时必须在后续多轮请求里把已进入当前请求的 assistant `reasoning_content` 按协议带回去，避免 DeepSeek 多轮工具调用报 400。这里回传的是 provider 协议元数据，不是长期记忆，也不能扩大当前上下文范围。

## 验收标准

当前专项以这些结果作为验收：

- GPT-5.4 走正式适配链路，不借临时分支“顺带兼容”
- `responses` 和 `chat.completions` 的职责边界清楚
- `.deadmouse/.env` 是唯一正式运行配置入口
- `doctor` 与真实请求链路使用同一配置判断
- DeepSeek 官方 V4 不再依赖旧 `chat/reasoner` 模型名表达思考模式
- DeepSeek 官方 V4 对已进入请求的 assistant 消息保留协议必要的 `reasoning_content`，reasoning 不进长期记忆
- 已完成真实请求和真实 Agent 回合验证

## 当前定稿

当前仓库已经进入“GPT-5.4 正式接入”的定稿状态。模型接入不再围绕单一 provider 生长，而是以可替换、可验证、可继续扩展的 Provider 适配边界为准。
