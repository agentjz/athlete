# 工具体系规划与收敛

## 这份文档回答什么问题

这份文档说明 Deadmouse 当前的工具体系已经收敛成什么样，以及为什么当前工具面不再继续发散。

## 当前目标与状态

当前工具专项已经完成以下收口：

- 保留现有正式工具命名
- 不增加平行工具族
- 保持正式工具默认全量可见
- 增加 `find_files`
- 收紧 `edit_file`
- 增强 `read_file`
- 增强 `search_files`
- 轻量化 `list_files`

当前结论已经明确：Deadmouse 借鉴外部项目时，借的是工具设计和约束，不是整套工具壳子。

## 适用范围

当前收口覆盖的是：

- 文件探索工具
- 文件写入与精修工具
- 运行时的正式工具面
- 模型选择工具时的主要路径

这里的“收敛”指不再制造平行工具入口，不指按阶段隐藏工具，也不指让机器层替模型选择工具。

## 范围外事项

当前明确不做：

- 为同一能力再造一套平行工具名
- 把 shell 别名直接搬成正式工具名
- 把外部项目的整套工具框架搬进仓库
- 按阶段裁剪模型可见工具集合
- 让治理元数据变成机器代选工具的依据

## 核心场景

当前工具面按场景已经清楚分工：

- 看目录，用 `list_files`
- 按路径模式找文件，用 `find_files`
- 按内容查找，用 `search_files`
- 读取文件正文，用 `read_file`
- 小范围精修已有文件，用 `edit_file`
- 需要 patch 语义时，用 `apply_patch`

其中 `run_shell` 仍然保留，但不再承担文件探索或普通读写的默认入口。
`run_shell` 当前补齐了结构化 runtime 结果：状态、超时/中断标记、输出截断标记和落盘路径。

## 关键约束

当前工具体系坚持以下约束：

- 同一能力只保留一条正式入口
- 写工具的约束强度高于只读工具数量
- 工具名优先表达动作和对象，不为“更短”而改名
- 工具仍然通过统一 registry、governance 和宿主注入边界接入
- 写工具统一把 `changedPaths`、diff、diagnostics、session diff 收口到正式结果
- 工具排序只能改变展示优先级，不能减少默认可见工具集合
- 工具治理只服务执行边界、结果信号和收口证据，不替 Lead 做策略选择
- 网页工具排序不再塑造“浏览器优先”的唯一正确路线；轻量 HTTP、下载和浏览器能力都应可见，由模型按任务判断
- `load_skill` 是显式加载入口，不是机器层替模型选择路线的硬门

## 验收标准

当前专项按下面的结果验收：

- `find_files` 已成为正式的路径模式查找工具
- `edit_file` 已要求正式 identity 与锚点
- `read_file` 已具备 continuation 语义
- `search_files` 已具备更结构化的搜索参数
- `run_shell` 已具备结构化执行状态与长输出截断落盘
- 工具优先级调整不会减少默认工具集合
- 工具优先级调整不会把技能加载、浏览器、后台或队友通道变成默认强制路线
- closeout 的局部防打转过滤不会外溢到普通执行阶段
- 当前工具面没有长出 `read / write / edit / bash / grep / ls` 这类平行入口

## 当前定稿

Deadmouse 当前的工具体系已经进入“少而准、边界清楚、主路径稳定”的定稿状态。重点不再是继续加名字，而是让现有正式工具更可靠、更可治理。
