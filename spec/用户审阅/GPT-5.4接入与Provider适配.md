# GPT-5.4 接入与 Provider 适配

## 1. 这个项目这一版到底要做什么

这一版不是单纯“把一个新模型名填进配置里”，而是先把 Athlete 当前和 DeepSeek 耦合过深的模型接入方式整理成可长期演进的正式结构，再把 GPT-5.4 作为第一批干净接入对象接进去。

这一版要完成的事情只有三件：

- 把模型接入边界拆成两层：一层是与 provider 无关的通用请求协议层，另一层是 provider / model-specific 的能力适配层。
- 让 GPT-5.4 通过当前可用的 OpenAI 中转接口真实跑通，而不是继续借 DeepSeek 逻辑“顺带兼容”。
- 把当前临时认证来源收敛进项目自己的 `.athlete/.env`，让运行真相源回到 Athlete 配置系统本身。

## 2. 这一版为什么值得做

如果这一步不先做，Athlete 后面每多接一个模型，就会多长一层 provider 特判，主循环、请求链路、恢复逻辑和配置入口都会越来越脏。

这一版值得做，不是因为 GPT-5.4 本身更“高级”，而是因为它会逼着系统回答一个更根本的问题：Athlete 到底是在做一个可替换模型底座，还是在做一个带很多 DeepSeek 残留假设的单厂商壳子。

这一步做对以后，后面继续接入别的 OpenAI-compatible 模型、保留 DeepSeek、替换中转站，都会容易很多。

## 3. 这一版是给谁用的

- 项目 Owner：需要看到这次改造到底解决什么问题，边界在哪里，验收怎么判断。
- 后续实现 Agent：需要在不猜需求的前提下完成 adapter 重构、GPT-5.4 接入和验证。
- 后续维护者：需要在继续接更多 provider 时，有清楚的结构边界可以沿着长，而不是再把逻辑塞回主流程。

## 4. 核心场景和主流程

核心场景是：Owner 希望 Athlete 不再默认围着 DeepSeek 长，而是可以把 GPT-5.4 通过当前可用的 OpenAI 中转入口真实接进来，并且后面还能继续接别的模型。

主流程应当是：

1. 在 `.athlete/.env` 中配置当前要使用的 provider、base URL、model、API key，以及必要的请求策略。
2. `doctor` 用真实配置探测 provider 是否可达，并且对中转站的慢响应保持更宽松的超时容忍。
3. 运行时根据配置选择正确的 provider 适配层，而不是在核心请求链路里写死 DeepSeek 分支。
4. GPT-5.4 通过当前实际可跑通的协议完成一次真实请求。
5. Athlete 再基于同一条正式接入链路完成一次真实 Agent 任务回合，而不是只做裸 API 探测。

## 5. 验收标准

这一版只有在下面这些条件同时满足时，才算完成：

- Provider 接入已经明确分成“通用请求协议层”和“能力适配层”两层，DeepSeek 逻辑不再污染通用层。
- GPT-5.4 不是靠复用 DeepSeek 的特判跑起来的，而是通过独立、清楚、正式的 provider 适配逻辑接入。
- Athlete 运行时不再依赖 `TT-config auth` 作为正式配置入口；最终配置真相源是项目自己的 `.athlete/.env`。
- `doctor` 可以基于真实配置完成连通性检查，并且不会因为中转站偏慢就过早判定失败。
- 至少完成一次真实 Responses API 请求验证，确认 GPT-5.4 在当前入口上可实际返回结果。
- 至少完成一次真实 Athlete Agent 任务回合验证，不是只 ping 接口；该回合必须证明模型接入链路真实可用。

## 6. 范围外事项

这一版明确不做这些事：

- 不追求一次性把所有 provider 都重构到最完美状态。
- 不为了“统一外观”去强行抹平所有模型的细节差异。
- 不在这一轮顺手改造无关宿主、无关通道或无关产品功能。
- 不为了兼容旧残余行为而牺牲新的 provider 边界正确性。

## 7. 成熟参考与可复用对象

这一版不是从零造轮子，已有可直接复用或改造的对象包括：

- 仓库里已经存在的 OpenAI SDK 接线能力。
- 仓库里现有的配置系统、`.athlete/.env` 入口和 `doctor` 命令。
- OpenAI-compatible provider 这条现有接入方向本身。
- 当前项目里已经存在的 provider capability、错误恢复、observability 和 turn 执行链路。

这次要做的不是推翻这些已有能力，而是把它们重新整理到更清楚的边界里。

## 8. 待确认事项

- 这一轮是否要求在重构过程中完整保留当前 DeepSeek 行为不变，还是允许先以新结构和 GPT-5.4 跑通为优先目标。
- 真实 Agent 验收回合具体用哪个任务最合适，应在开始实现前定一个稳定、可重复的最小验证任务。
