# 宪法原则

这一组文档是给人类审阅项目方向用的根本原则。

它们回答的不是“某个模块怎么写”，而是：

- 这个项目为什么要这样长
- 哪些边界不能被实现细节反推翻
- 当代码、测试和文档冲突时，应该先保护什么

## 现在怎么读

### 01 到 16

这部分是当前已经稳定成立的底层原则：

- 单智能体主循环
- 工具处理器边界
- 计划、知识、上下文、任务图
- 后台、队友、协议、工作区
- session、provider、配置

### 17 到 25

这部分是当前工程已经必须正面处理的系统原则：

- 扩展口怎么长
- 主循环和文件怎么保持清醒
- 失败测试、验证、证据、机器状态
- 文本链路、错误兼容、旧残余清理

## 使用规则

1. 宪法原则只写长期有效、能跨多个模块成立的规则。
2. 一次性实现技巧不要升格成宪法。
3. 模块细节、接口细节和仓库流程写去 `技术实现/`。
4. 当产品定位发生变化时，要先检查这里是否需要更新。
