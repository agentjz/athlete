# 17-扩展靠事件生长

## 原则

扩展不能只靠往 system prompt 里继续堆规则。

系统要通过清晰的扩展口、运行时事件点和生命周期边界来生长。

## 为什么

Deadmouse 现在已经不是一个单兵型 Agent。

它后面还会继续长出：

- 更多技能
- 更多工具
- 更多角色
- 更多宿主
- 更多运行时通知和可见事件

如果没有正式扩展点，最后会出现三种坏事：

1. 主循环越来越胖
2. 宿主越来越会偷偷拼 runtime
3. 新能力只能靠 prompt 和胶水代码硬塞进去

## 在 Deadmouse 里的含义

Deadmouse 当前正式扩展口至少有四类：

1. 工具扩展
   - tool registry
   - governed tool metadata
2. 技能扩展
   - skill catalog
   - skill indexing / explicit loading
3. 外部能力扩展
   - MCP
4. 运行时与宿主扩展
   - runtime callbacks
   - turn visible events
   - lifecycle closeout
   - host extra tools
   - host identity

也就是说，Deadmouse 现在已经不只需要“tool / skill / MCP”三个扩展口。

它还必须明确：

- turn 开始点
- turn 执行点
- pause / abort / error / finalize 收口点
- visible event 输出点
- 宿主注入 extra tool 的边界

这些 runtime / lifecycle 点，必须是正式边界，不是散装胶水。

## 当前对应

- `src/capabilities/tools/core/runtimeRegistry.ts`
- `src/capabilities/skills/`
- `src/capabilities/mcp/`
- `src/host/`
- `src/chat/`
- `src/ui/`

## 当前要求

1. 新扩展优先长在正式扩展口，不优先长在 prompt 里。
2. 宿主扩展必须走 `src/host/`，不能绕进核心内部模块。
3. runtime 事件和生命周期收口要作为正式设计对象，而不是“以后再补”。
