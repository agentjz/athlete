# 项目范围

## 这份文档回答什么问题

这份文档说明 Kitty 当前已经正式纳入哪些能力，以及哪些事项明确不纳入当前项目范围。

## 当前范围

### 1. 耐跑底盘

当前已经纳入：

- 会话现场（session）作为任务现场
- checkpoint、continuation 和恢复路径
- 上下文压缩与压缩后续跑
- verification、acceptance 和 closeout
- runtime stats 与运行摘要

### 2. Lead 决策协议

当前已经纳入：

- Lead 的当前目标 frame
- `lead / teammate / subagent / background` 的正式能力入口
- 显式 assignment 与机器硬约束（机器只校验事实边界，不评估委派必要性）
- 委派后回主复核屏障（禁止同链路连续无边界深挖）
- `ready / blocked / active / merge / wait` 等状态事实

### 3. 执行边界与预算事实

当前已经纳入：

- 执行通道的 runtime、idle、终止和并发边界
- provider recover、managed turn、subagent 等结构化预算事实
- 预算命中只产生事实、closeout 或 wake signal，不替 Lead 选择路线
- 任一硬边界命中都必须强制返回 Lead

### 4. 控制面

当前已经纳入：

- SQLite 控制面账本
- 任务、队友、执行、协议和 worktree 的正式状态
- 等待、合流和硬边界检查所依赖的持久化事实

### 5. 扩展与宿主

当前已经纳入：

- tools
- skills
- MCP
- CLI
- Telegram 私聊

## 范围外事项

当前明确不在范围内：

- 面向大众的图形化编排平台
- 商业化的 swarm 市场或 agent 广场
- 跨项目长期人格注入
- 没有控制面约束的自治 agent 军团
- 为了热闹而默认放出大量并发 agent
- 仅凭关键词的单点委派触发语义
- 用户档位或启动参数驱动的机器自动派发语义

## 范围判断标准

当前项目用下面两条判断一个需求是否应进入主线：

- 是否增强耐跑能力、Lead 活判断能力、控制面能力或正式扩展口
- 是否会破坏现有真相源、宿主边界或统一主路径

如果一个需求不能增强主线能力，反而会削弱这些边界，就不进入当前范围。

## 验收标准

范围内能力要满足以下要求：

- 能落在正式真相源上
- 能进入统一主路径
- 能说明边界与范围外事项
- 能通过自动测试或真实链路验证
- 委派必须来自 Lead 的显式入口，通过机器硬约束检查，且可被结构化审计
- subagent 超限必须有结构化 reason，并回主复核

## 当前定稿

Kitty 当前的范围已经收口在“耐跑底盘 + Lead 决策协议（显式入口/机器硬约束/执行边界/回主复核） + 控制面账本 + 正式扩展口 + 多宿主接入”这条主线上，不再以“功能越多越好”或“机器自动派发更多执行体”为目标。
