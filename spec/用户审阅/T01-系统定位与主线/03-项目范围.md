# 项目范围

## 这份文档回答什么问题

这份文档说明 Deadmouse 当前已经正式纳入哪些能力，以及哪些事项明确不纳入当前项目范围。

## 当前范围

### 1. 耐跑底盘

当前已经纳入：

- 会话现场（session）作为任务现场
- checkpoint、continuation 和恢复路径
- 上下文压缩与压缩后续跑
- verification、acceptance 和 closeout
- runtime stats 与运行摘要

### 2. 总指挥能力

当前已经纳入：

- Lead 的任务预处理
- 最小任务图与任务拆分
- `lead / teammate / subagent / background` 的正式路由
- 委派评估与机器硬约束（模型必要性评估只做建议留痕，机器策略闸门只卡事实边界）
- 委派后回主复核屏障（禁止同链路连续无边界深挖）
- `ready / blocked / active / merge / wait` 的机器语义

### 3. 委派档位与预算控制

当前已经纳入：

- 用户可见档位：`快 / 均衡 / 深度`
- 档位影响委派评估倾向、并发委派上限和 subagent 预算
- subagent 硬预算：工具数、模型轮次、墙钟时长
- 任一预算超限都必须强制返回 Lead

### 4. 控制面

当前已经纳入：

- SQLite 控制面账本
- 任务、队友、执行、协议和 worktree 的正式状态
- 调度、等待、合流所依赖的持久化裁决信息

### 5. 扩展与宿主

当前已经纳入：

- tools
- skills
- MCP
- CLI
- Telegram 私聊

## 范围外事项

当前明确不在范围内：

- 面向大众的图形化编排平台
- 商业化的 swarm 市场或 agent 广场
- 跨项目长期人格记忆
- 没有控制面约束的自治 agent 军团
- 为了热闹而默认放出大量并发 agent
- 仅凭关键词的单点单点委派触发语义

## 范围判断标准

当前项目用下面两条判断一个需求是否应进入主线：

- 是否增强耐跑能力、总指挥能力、控制面能力或正式扩展口
- 是否会破坏现有真相源、宿主边界或统一主路径

如果一个需求不能增强主线能力，反而会削弱这些边界，就不进入当前范围。

## 验收标准

范围内能力要满足以下要求：

- 能落在正式真相源上
- 能进入统一主路径
- 能说明边界与范围外事项
- 能通过自动测试或真实链路验证
- 委派必须有显式入口，经过 evaluator 建议记录，并通过 policy gate 的机器硬约束检查，且可被结构化审计
- subagent 超限必须有结构化 reason，并回主复核

## 当前定稿

Deadmouse 当前的范围已经收口在“耐跑底盘 + 总指挥调度（显式入口/评估留痕/机器硬约束/硬预算/回主复核） + 控制面账本 + 正式扩展口 + 多宿主接入”这条主线上，不再以“功能越多越好”为目标。
