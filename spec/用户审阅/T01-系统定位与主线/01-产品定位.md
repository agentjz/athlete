# 产品定位

## 这份文档回答什么问题

这份文档说明 Kitty 当前是什么产品、主要服务谁，以及项目判断取舍时以什么价值为准。

## 当前定位

Kitty 当前不是“在终端里聊天的 AI 员工”。

当前定稿定位是：

**一个以 Lead（主 Agent）为唯一活判断者、以机器事实边界和正式能力协议为底座的耐跑 agent harness。**

这个定位强调三件事：

- 它要能持续推进长任务，而不是只完成一次对话
- 它要把策略判断留给 Lead，而不是让机器层变成第二个大脑
- 它要能承接工具、技能、MCP 和多宿主接入，但所有能力只作为可见选项和显式执行通道存在

## 当前目标与状态

当前项目已经围绕下面的目标落地：

- 让 Lead 能基于正式任务状态决定自执行、委派、等待和合流
- 让会话现场和控制面账本承担正式真相源，而不是让宿主自己保存运行真相
- 让工具、技能、MCP 和宿主都通过统一边界接入
- 让长任务具备验证、续跑、恢复和收口能力

这说明 Kitty 当前的主线不是“更像聊天产品”，也不是“机器自动派发更多 agent”，而是“让 Lead 在可记录、可恢复、可验证的机器边界内长期工作”。

## 适用范围

Kitty 当前主要适用于：

- 需要多步自动化和连续推进的任务
- 需要整理文件、文档、网页和外部资料的工作
- 需要 Lead 显式选择并回收多条执行路径的场景
- 需要把能力接入现有 Agent 系统、但不想维护多套运行底座的场景

## 范围外事项

Kitty 当前明确不把自己定义为：

- 纯聊天机器人
- 只靠单个 Agent 完成全部工作的单兵系统
- 默认放出大量 agent 并发的 swarm 表演系统
- 拖拽式工作流平台
- 先堆功能、后补真相源和边界的杂糅系统

## 审阅重点

审阅能力时，当前以这几个问题作为判断标准：

- 是否增强了 Lead 的活判断能力和证据可见性
- 是否增强了耐跑、恢复、验证和收口能力
- 是否守住了控制面账本、会话现场和宿主边界
- 是否继续维持统一扩展口，而不是长出平行入口

## 当前定稿

当前仓库的文档、实现和测试已经共同指向同一结论：Kitty 的正式定位是“Lead 活判断 + 机器事实边界 + 正式能力协议 + 耐跑恢复”的 agent harness，而不是聊天外壳、机器派发器或功能堆叠系统。
