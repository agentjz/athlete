# 测试策略

## 目标

测试优先保护 Athlete 的骨架，
不是为了表面覆盖率。

本轮开始，控制面测试还要额外保护：

- SQLite 账本初始化
- 重载后一致性
- 并发冲突裁决
- 旧 JSON 真相源退场
- 调度 split / dispatch / wait / merge / recover
- CLI fast path / lightweight path / full runtime path 边界
- config schema version 与错误配置 fail-closed 行为
- 用户可读错误分类与 runtime summary 产品文本链路

## 当前重点

- 主循环没坏
- context / continuation / recovery 没坏
- lightweight context externalization 没坏
- checkpoint / reload / resume 没坏
- runtime stats / runtime summary 没坏
- CLI 启动链没有重新耦回重初始化
- SQLite 控制面账本没坏
- 多 Agent 调度生命周期没坏
- task / background 没坏
- team / protocol / policy 没坏
- worktree isolation 没坏
- orchestrator 仍只读取统一真相
- skill catalog 与 skill loading 没坏
- Playwright MCP 接线、优先级与 artifact 路由没坏
- provider capability / fallback / reasoning 行为没坏
- browser capability workflow guard / priority 没坏
- document capability / acceptance signal 没坏
- one-shot closeout contract 没坏
- interactive quit guard 不会漏掉运行中的后台进程
- post-completion closeout 不会退回无意义循环

## 当前测试类型

### 1. 骨架测试

保护架构约束，不让主骨架被改坏。

### 2. 缺陷回归测试

真实 bug 修掉后，把 bug 固化成测试。

### 3. 结构预算测试

保护主循环和关键控制面文件不继续长胖。

### 4. 账本一致性测试

保护控制面单一真相源：

- 干净初始化不依赖旧 JSON
- reload 后 task / team / protocol / background / worktree 一致
- 同一任务不能被两个主体同时 claim 成功
- worktree 绑定不会因并发或重载失真
- orchestrator 不会读到并行文件残影
- 旧 JSON 不会继续被写入或双写

### 5. 调度生命周期测试

保护正式调度器，而不是 prompt 协调：

- 复杂 objective 会拆成最小任务图
- 同一 child work 不会被重复 dispatch
- teammate 不会重复 claim 已被占用任务
- background child 完成后会推进 merge / next lead step
- 没有 lead-ready work 时会进入 wait
- 子任务完成后父任务能 join 或完成
- reload 后能恢复 active owner / wait reason / next action
- 旧残余调度路径不会继续参与裁决

### 6. CLI 产品工程测试

保护用户直接感知到的启动、配置、报错和透明度：

- `--version` / `version` 是真实 fast path
- `--help` 不会被 runtime 初始化或缺 key 污染
- `config path` 在坏配置场景下仍然可用
- `config show / get / set` 输出稳定，且 schema version 规则明确
- `doctor` 会区分用户可修复错误、环境 / 网络问题、provider 响应问题
- one-shot closeout / `/runtime` / visible events 文本稳定可读
- Windows 文本链路不因默认代码页退化

## 命令

默认本地验证：

```powershell
npm.cmd run check
npm.cmd test
```

只做测试编译：

```powershell
npm.cmd run test:build
```

核心测试集：

```powershell
npm.cmd run test:core
```

真实 API 验证（不进入默认 `npm.cmd test`）：

```powershell
npm.cmd run verify:runtime-context-api
npm.cmd run verify:runtime-checkpoint-api
npm.cmd run verify:runtime-observability-api
```

## 当前原则

1. 骨架测试优先。
2. 回归测试优先。
3. fail-first 优先，先把新约束写成失败测试。
4. 默认测试不要强依赖不稳定外部 API。
5. 真实 API / 真实浏览器验证可以做，但不要强塞进默认 `npm.cmd test`。
6. 账本类能力必须同时覆盖：
   - 干净初始化
   - reload 后一致性
   - 事务冲突裁决
   - 旧真相源退场
7. 调度类能力必须同时覆盖：
   - split
   - dispatch
   - wait
   - merge
   - recover
   - legacy-path exclusion
8. 生态去耦类测试要优先保护“上层不认生态字符串，只认 capability / signal / transition”。
9. CLI 产品工程类测试要优先保护“轻命令不拉重 runtime、坏配置不静默继续、错误必须可操作”。

## 不建议的做法

- 为了数量堆很多低价值小测试
- 把不稳定的真实 API 测试塞进默认 `npm.cmd test`
- 只测 prompt 文案，不测底层机器状态
- 让旧 JSON 路径继续留在测试断言里，变相保活旧结构
