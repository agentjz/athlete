# 执行协议平台

## 目标

执行协议平台是 Deadmouse 的通用执行底座。

它不定义具体能力，不决定是否派发，也不替 Lead 拆任务。它只定义所有执行能力进入和返回 Lead 的统一协议。

## 运行时原则

1. Lead 是决策者。
2. 机器层只执行 Lead 的显式动作。
3. team、subagent、workflow、skill、tool、MCP、external agent、plugin、background 都只是协议平台的接入者。
4. 能力默认可用，但可用不等于意图。
5. 唤醒信号只是门铃，不是真相来源。

## 通用协议链路

```text
Capability -> Assignment -> Execution -> Progress -> Artifact -> Closeout -> WakeSignal -> Lead
```

### Capability

说明一个能力是什么、适合什么、不适合什么、输入输出协议是什么、预算成本是什么。

它只提供选择信息，不产生派发意图。

### Capability Package

能力包是所有生态接入的统一描述。

它必须表达 id、kind、name、description、version、协议版本、输入输出契约、runner 类型、所需工具、预算策略、artifact 策略、closeout 策略、来源、是否内置、Lead 选择前需要知道的信息，以及机器层被允许做什么。

能力包不是派发意图。注册了能力包，只代表 Lead 可以看见这个选项。

外部生态可以先提供 Capability Manifest。Manifest 会被规范化成 Capability Package；规范化只校验和适配字段，不读取执行环境，不选择策略，不启动 runner。

多个 Manifest 可以组成 Manifest Bundle。Bundle 会变成一个 Capability Adapter，再进入 Registry。这样外部插件包、agent catalog、skill pack、MCP pack、workflow pack 都可以通过同一个入口接入；核心不为任何具体生态写专用分支。

### Capability Registry

Registry 统一收集所有能力包。

Lead 只看低噪声 summary，不关心能力来自 team、subagent、workflow、skill、tool、MCP、external agent、plugin 还是 background。

Registry 只暴露能力，不自动路由，不自动选择，不替 Lead 产生 intent。

### Capability Adapter

Adapter 只负责把某个来源转换成能力包。

skill、tool、workflow、teammate、subagent、MCP、external agent、plugin 都必须通过 adapter 或等价 package provider 接入。Adapter 不做策略判断，不决定是否使用。

### Capability Runner

Runner 只执行 Lead 已通过 Assignment 选择的能力。

Runner 可以创建 Execution、启动执行、记录 Progress、记录 Artifact、写 Closeout、发 WakeSignal、fail closed，但不能判断该不该使用该能力。

### Assignment

Lead 派发时必须给出的标准派工单。

字段包括目标、范围、约束、期望输出、预算和回流条件。

### Execution

机器层创建的执行事实记录。

它记录谁派的、谁执行、什么状态、什么时候开始结束、关联哪些 artifact。

### Progress

执行过程中的事实事件。

它可以记录进度、artifact、warning、blocked、heartbeat，但不替 Lead 做战略总结。

### Artifact

执行过程产生的稳定证据引用。

它可以指向文件、命令结果、日志、观察记录或 execution 记录。Artifact 只保存证据位置和说明，不替 Lead 判断结论。

### Closeout

执行体最终交回 Lead 的标准报告。

字段包括状态、摘要、证据、改动路径、验证、风险和下一步建议。

### WakeSignal

执行完成、失败、暂停、预算耗尽或中止时写入的单文件唤醒信号。

它只负责唤醒 Lead；Lead 必须读取 execution 记录、closeout、artifact 和验证结果后再判断。

## 通用与定制边界

- `src/protocol/` 只放通用协议。
- `src/capabilities/` 收拢所有能力生态入口。
- `src/capabilities/team/` 只放 teammate 能力生态。
- `src/capabilities/subagent/` 只放 subagent 能力生态。
- `src/capabilities/workflows/` 只放 workflow 能力生态。
- `src/capabilities/skills/` 放 skill 运行时，`src/capabilities/skills/packages/` 放内置 skill 包。
- `src/capabilities/tools/core/` 放 tool 注册、治理、执行框架。
- `src/capabilities/tools/packages/` 放具体 tool 包。
- `src/capabilities/mcp/`、`src/capabilities/background/` 只能通过 adapter / package provider 接入通用协议。
- `src/capabilities/registry.ts` 是 Lead 面向能力生态的统一装配口。
- `src/protocol/` 不导入 `src/capabilities/`，也不硬编码任何具体生态名。
- 未来定制能力必须注册为 Capability Package，并遵守 Assignment、Execution、Progress、Artifact、Closeout、WakeSignal。

## 禁止事项

- 禁止机器层因为文本复杂而自动派发。
- 禁止机器层因为某能力可用而自动创建执行体。
- 禁止 Registry 或 Adapter 替 Lead 选择能力。
- 禁止 Runner 在没有 Assignment 的情况下启动能力。
- 禁止定制能力绕过 AssignmentContract。
- 禁止执行体绕过 CloseoutContract。
- 禁止把 WakeSignal 当成结果真相。
