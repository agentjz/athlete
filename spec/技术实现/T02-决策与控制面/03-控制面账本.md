# 控制面账本

## 文档目的

本文说明 Deadmouse 当前的控制面账本负责哪些实体、落在什么存储中，以及它与会话现场之间怎样分工。

## 模块目标与当前状态

控制面账本当前是项目级正式裁决层，负责跨任务、跨执行和跨 worktree 的状态。

## 正式边界

主要代码边界：

- `src/control/ledger/open.ts`
- `src/control/ledger/migrations.ts`
- `src/control/ledger/taskRepo.ts`
- `src/control/ledger/teamRepo.ts`
- `src/control/ledger/requestRepo.ts`
- `src/control/ledger/policyRepo.ts`
- `src/control/ledger/executionRepo.ts`
- `src/control/ledger/worktreeRepo.ts`

## 真相源与状态归属

当前 ledger 负责回答：

- 有哪些 task
- 谁在做、谁可用、谁已 shutdown
- 哪些协议请求与协同策略仍有效
- 哪些 execution 正在运行、已暂停或已完成
- 哪些 worktree 可用、已保留或已移除

当前分工原则：

- session 现场负责单次任务现场
- control plane ledger 负责跨任务裁决与协同状态

## 主路径

当前主路径如下：

1. 通过 `open.ts` 打开项目级 SQLite ledger。
2. repo 层读写 task、team、request、policy、execution、worktree。
3. agent、execution、tasks 和 worktree 模块统一从 repo 层读取正式状态。
4. migration 与 layout 负责 ledger schema 的初始化和升级。

## 模块职责与非职责

### ledger 负责

- 项目级裁决状态
- task / team / execution / worktree 等实体持久化
- 为执行、等待和显式工具动作提供统一查询结果

### ledger 不负责

- 单个 session 内的消息现场
- 宿主展示缓存
- observability 旁路日志

## 失败路径与异常路径

当前明确处理：

- 控制面状态优先进入 SQLite，而不是先发明 JSON 真相
- session 与 ledger 若同时裁决同一件事，视为边界错误，需要继续收敛
- 宿主层和展示层不能成为控制面真相

## 当前落地决定

当前控制面账本已经承担项目级裁决职责。增加协同状态应优先进入 ledger，而不是继续在宿主或临时 JSON 中分散落地。
