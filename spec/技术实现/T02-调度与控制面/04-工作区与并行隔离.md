# 工作区与并行隔离

## 文档目的

本文说明 Deadmouse 当前如何用 worktree 为 task-bound execution 提供隔离目录，以及 worktree 与 task、execution 的关系。

## 模块目标与当前状态

当前 worktree 是共享隔离层，不是 teammate 私有机制。它服务的是并行执行安全和 task 级目录绑定。

## 正式边界

主要代码边界：

- `src/worktrees/store.ts`
- `src/worktrees/ledger.ts`
- `src/worktrees/events.ts`
- `src/execution/taskBinding.ts`
- `src/control/ledger/worktreeRepo.ts`

## 真相源与状态归属

worktree 当前的正式状态包括：

- `active`
- `kept`
- `removed`

这些状态由 `WorktreeLedgerRepo` 记录在项目 ledger 中；事件日志只做旁路记录，不反向裁决状态。

task、execution 和 worktree 的职责当前分工如下：

- task：定义做什么、依赖谁、是否完成
- execution：定义谁在执行、执行到哪一态、是否已 closeout
- worktree：定义这项工作在哪个目录执行，以及目录是否仍有效

## 主路径

当前主路径如下：

1. task-bound execution 绑定 `taskId`。
2. `prepareExecutionTaskContext` 统一完成 claim 与 worktree attach。
3. worker 在 worktree cwd 中执行。
4. closeout 后，task、execution 和 worktree 各自按正式状态收口。
5. 创建新 worktree 前先执行 git worktree prune，清掉 git 里“目录已丢失但仍登记”的死 worktree。
6. `reconcile()` 会把丢失实体目录的 worktree 收敛到 `removed`，并解绑失效 task 关联。

## 失败路径与异常路径

当前明确处理：

- 无法创建 worktree 时，task claim fail closed
- `removed` worktree 不再被视为可用执行目录
- task 绑定的 worktree 丢失后，会在 reconcile 中解绑
- git 仍登记 missing worktree 时，创建路径先 prune 再重试主路径，不把死登记暴露成队友失败
- `completeTask` remove 会同时完成对应 task 并解绑 worktree

## 测试与验证

当前主要由以下测试保护：

- `tests/worktree-isolation.test.ts`
- `tests/control-plane-ledger.test.ts`
- `tests/execution-lanes.test.ts`

## 当前落地决定

当前并行隔离已经固定为“task / execution / worktree 三者绑定”的结构。任何并行执行能力都应沿这条主路径工作，而不是各自维护私有目录语义。
