# 多智能体调度

## 文档目的

本文描述 Deadmouse 当前的执行通道模型，说明 Lead 如何与 `subagent / teammate / background` 协同，以及执行状态如何被正式记录。

## 模块目标与当前状态

Deadmouse 当前不再维护三套互不相干的执行生命周期，而是统一为：

- `agent` lane
- `command` lane

其中：

- `subagent` 和 `teammate` 是 `agent` lane 的两种 profile
- `background` 是 `command` lane 的 profile

## 真相源与状态归属

执行状态当前以 execution record 为正式观察对象，主要状态包括：

- `queued`
- `running`
- `paused`
- `completed`
- `failed`
- `aborted`

Lead 在等待委派工作时，当前看的是 execution record 和相关 task 状态，而不是宿主私有循环或内存特例。

## 主路径

当前主路径如下：

1. orchestrator 决定任务落到哪条 lane/profile。
2. 委派候选动作先做 evaluator 建议评估，再过 policy gate 硬约束闸门。
3. `dispatch` 创建 execution record。
4. 异步执行统一通过 `__worker__ run --execution-id <id>` 拉起 worker。
5. worker 领取 task、worktree 和 session 上下文。
6. subagent 执行期间受硬预算约束（工具/轮次/时长），超限即强停并写结构化 reason。
7. 完成后通过 `closeExecution(...)` 写入正式 closeout。
8. `execution_closeout` 消息回到 Lead inbox。
9. 委派结束后回主屏障生效：必须先回 Lead 复核，再决定是否继续委派。
10. `progress` 与 `route` 再决定 wait、merge 或继续派发。

对 `background` profile，`background_terminate` 也走同一条 closeout contract：终止动作会把 execution 正式收口为 `aborted`，并继续通过 `execution_closeout` 回传，而不是绕开 execution record。

## 模块职责与非职责

### 执行通道负责

- 保存执行状态
- 连接 task、worktree、session 和 closeout
- 让 Lead 以统一方式等待和合流

### 执行通道不负责

- 再次定义任务规划
- 在宿主层各自长出私有 worker 协议
- 另造平行 lifecycle

## 失败路径与异常路径

当前明确处理：

- stale teammate / dead worker 会被 reconcile 关闭
- queued worker 长时间未获得 live pid 会失败关闭
- inline execution 在宿主退出后不会假装仍然运行
- 损坏的 persisted session 会 fail closed，不会偷偷重建
- 已完成 execution 不允许再次 start
- running background execution 被显式 terminate 时，必须通过同一 closeout 协议进入 `aborted`
- subagent 超过工具/轮次/时长任一预算时，必须强制返回 Lead，不能继续深挖
- 单链路委派后若未完成主控复核，不允许连续再次委派

## 当前落地决定

当前执行模型已经固定为“统一 lane + profile + execution record”。增加执行能力应继续作为 lane 或 profile 的扩展进入，而不是再造额外的执行制度。

后台 terminate、stale reconcile 和 worker closeout 现在都继续收敛到同一 execution lifecycle contract，不允许宿主旁路改状态。
