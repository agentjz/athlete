# Multi-Agent Scheduling

## Document Purpose

This document describes the current execution lane model: how Lead coordinates protocol-declared execution work, and how runtime state is recorded.

## Current Model

Deadmouse uses one execution lifecycle with two lanes:

- `agent` lane
- `command` lane

Profiles:

- `subagent` and `teammate` are `agent` profiles.
- `background` is a `command` profile.
- `workflow` is an `agent` profile.

## Truth Source And State Ownership

Execution state is owned by execution records. Primary statuses are:

- `queued`
- `running`
- `paused`
- `completed`
- `failed`
- `aborted`

Lead waits on execution records whose `deadmouse.lead-wait-policy` snapshot declares `while_execution_active`. It does not own a private polling loop and it does not spend model turns checking execution status while nothing changed.

## Runtime Story

1. Lead decides whether to start an execution, which capability to use, names, roles, and assignment text.
2. The model calls an explicit tool; the machine validates deterministic invariants and creates worker-backed execution records through the shared assignment execution path.
3. Workers start through the shared `__worker__ run --execution-id <id>` entry.
4. Each execution records the `AssignmentContract`, `CapabilityPackage`, `deadmouse.execution-policy`, and `deadmouse.lead-wait-policy` snapshots used at creation time.
5. Each execution receives `deadmouse.execution-boundary` with runtime and idle limits.
6. Agent and command executions run until completion, real failure, user interruption, or execution boundary.
7. Boundary hits close the execution as `paused` with a structured reason and return control to Lead review.
8. Normal completion and failure also close through `closeExecution(...)`.
9. Execution closeout publishes an event and sends a message to Lead inbox.
10. Lead wakes, reads execution closeout and related facts, then decides the next move.
11. Wake signals are not truth sources; closeout records are.

## Responsibilities

Execution lane owns:

- durable execution status
- worker launch protocol
- runtime and idle boundaries
- closeout records
- closeout events

Lead owns:

- delegation judgment
- merge/review decisions after closeout
- next-move judgment

Subagent/team workers own:

- assigned execution work
- evidence collection
- closeout summary and artifacts

## Non-Responsibilities

Execution lane must not:

- decide task strategy for the model
- invent delegation without Lead tool calls
- create task-board stages from prompts, wake signals, checkpoints, or compaction
- stop subagents because of low tool-call or model-turn budgets
- treat wake signals as state truth

Lead must not:

- self-poll execution work with empty model turns
- declare completion without closeout evidence
