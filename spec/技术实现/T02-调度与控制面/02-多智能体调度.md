# Multi-Agent Scheduling

## Document Purpose

This document describes the current execution lane model: how Lead coordinates `subagent`, `teammate`, and `background` work, and how runtime state is recorded.

## Current Model

Deadmouse uses one execution lifecycle with two lanes:

- `agent` lane
- `command` lane

Profiles:

- `subagent` and `teammate` are `agent` profiles.
- `background` is a `command` profile.

## Truth Source And State Ownership

Execution state is owned by execution records. Primary statuses are:

- `queued`
- `running`
- `paused`
- `completed`
- `failed`
- `aborted`

Lead waits on execution records and execution closeout events. It does not own a private polling loop and it does not spend model turns checking delegated status while nothing changed.

## Runtime Story

1. Lead decides whether to delegate, how many workers to create, names, roles, and assignment text.
2. The machine validates policy gates and creates worker-backed execution records.
3. Workers start through the shared `__worker__ run --execution-id <id>` entry.
4. Each execution receives `deadmouse.execution-boundary.v1` with runtime and idle limits.
5. Subagent and teammate executions run until completion, real failure, user interruption, or execution boundary.
6. Boundary hits close the execution as `paused` with a structured reason and return control to Lead review.
7. Normal completion and failure also close through `closeExecution(...)`.
8. Execution closeout publishes an event and sends a message to Lead inbox.
9. Lead wakes, reads execution closeout and related facts, then decides the next move.
10. Wake signals are not truth sources; closeout records are.

## Responsibilities

Execution lane owns:

- durable execution status
- worker launch protocol
- runtime and idle boundaries
- closeout records
- closeout events

Lead owns:

- delegation judgment
- merge/review decisions after closeout
- next-step orchestration

Subagent/team workers own:

- assigned execution work
- evidence collection
- closeout summary and artifacts

## Non-Responsibilities

Execution lane must not:

- decide task strategy for the model
- invent delegation without Lead tool calls
- stop subagents because of low tool-call or model-turn budgets
- treat wake signals as state truth

Lead must not:

- self-poll delegated work with empty model turns
- declare completion without closeout evidence
