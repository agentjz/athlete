# 多智能体调度

## 文档目的

本文描述 Deadmouse 当前的执行通道模型，说明 Lead 如何与 `subagent / teammate / background` 协同，以及执行状态如何被正式记录。

## 模块目标与当前状态

Deadmouse 当前不再维护三套互不相干的执行生命周期，而是统一为：

- `agent` lane
- `command` lane

其中：

- `subagent` 和 `teammate` 是 `agent` lane 的两种 profile
- `background` 是 `command` lane 的 profile

## 真相源与状态归属

执行状态当前以 execution record 为正式观察对象，主要状态包括：

- `queued`
- `running`
- `paused`
- `completed`
- `failed`
- `aborted`

Lead 在等待委派工作时，当前看的是 execution record 和相关 task 状态，而不是宿主私有循环或内存特例。

agent lane 的入口不再由关键词或复杂度猜测打开。`teammate` 只由 `/team` 打开，`subagent` 只由 `/subagent` 打开，两者同时需要时由 `/team/subagent` 打开；没有这些前缀时，Lead 默认单兵执行。

## 主路径

当前主路径如下：

1. 用户用 `/team`、`/subagent`、`/team/subagent` 明确打开 agent lane；没有前缀时 Lead 默认单兵执行。
2. Lead 根据当前目标、账本事实和用户前缀决定是否进入 `subagent / teammate / background` 执行通道；机器不能靠关键词替 Lead 或用户猜测 agent lane。
3. 委派候选动作先做 evaluator 建议评估，再过 policy gate 硬约束闸门。
4. `dispatch` 或 Lead-only `task` 工具只负责创建 worker-backed execution record，不直接跑完子代理，也不直接判断父任务完成。
5. 异步执行统一通过 `__worker__ run --execution-id <id>` 拉起 worker。
6. worker 领取 task、worktree 和 session 上下文。
7. 每条 execution 创建时统一带 `deadmouse.execution-boundary.v1`；command lane 和 agent lane 都由机器层执行 runtime/idle 边界。
8. agent lane 到 boundary 时收口为 `paused`，写入结构化 reason，并把状态回传 Lead review；正常完成或失败也通过 `closeExecution(...)` 写入正式 closeout。
9. `execution_closeout` 消息回到 Lead inbox。
10. 委派结束后回主屏障生效：必须先回 Lead 复核，再决定是否继续委派。
11. `progress` 与 `route` 再决定 wait、merge 或继续派发。

对 `background` profile，`background_terminate` 也走同一条 closeout contract：终止动作会把 execution 正式收口为 `aborted`，并继续通过 `execution_closeout` 回传，而不是绕开 execution record。

## 模块职责与非职责

### 执行通道负责

- 保存执行状态
- 连接 task、worktree、session 和 closeout
- 让 Lead 以统一方式等待和合流

### 执行通道不负责

- 再次定义任务规划
- 在宿主层各自长出私有 worker 协议
- 另造平行 lifecycle
- 根据聊天关键词、复杂度或旧任务残余自行启动 teammate / subagent

## 失败路径与异常路径

当前明确处理：

- stale teammate / dead worker 会被 reconcile 关闭
- queued worker 长时间未获得 live pid 会失败关闭
- 旧 `inline` launch mode 已从 execution schema、类型和测试主路径删除；新建 execution 只接受 `worker`，不为旧子代理同步路径保活
- 损坏的 persisted session 会 fail closed，不会偷偷重建
- 已完成 execution 不允许再次 start
- running background execution 被显式 terminate 时，必须通过同一 closeout 协议进入 `aborted`
- subagent 超过工具/轮次/时长任一预算时，必须强制返回 Lead，不能继续深挖
- teammate / subagent agent execution 到 runtime 或 idle boundary 时，机器必须暂停 execution 并回 Lead review，不能只靠提示词自觉
- 没有 `/team`、`/subagent`、`/team/subagent` 前缀时，不允许新建 teammate / subagent agent execution
- 单链路委派后若未完成主控复核，不允许连续再次委派

## 当前落地决定

当前执行模型已经固定为“统一 lane + profile + execution record + execution boundary”。增加执行能力应继续作为 lane 或 profile 的扩展进入，并复用同一 boundary / closeout contract，而不是再造额外的执行制度。

后台 terminate、stale reconcile、worker closeout 和 agent boundary pause 现在都继续收敛到同一 execution lifecycle contract，不允许宿主旁路改状态。
