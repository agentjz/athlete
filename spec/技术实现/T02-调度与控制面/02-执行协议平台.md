# 执行协议平台

## 目标

执行协议平台是 Deadmouse 的通用执行底座。

它不定义具体能力，不决定是否派发，也不替 Lead 拆任务。它只定义所有执行能力进入和返回 Lead 的统一协议。

## 运行时原则

1. Lead 是决策者。
2. 机器层只执行 Lead 的显式动作。
3. team、subagent、workflow 都只是协议平台的接入者。
4. 能力默认可用，但可用不等于意图。
5. 唤醒信号只是门铃，不是真相来源。

## 通用协议链路

```text
Capability -> Assignment -> Execution -> Progress -> Closeout -> WakeSignal -> Lead
```

### Capability

说明一个能力是什么、适合什么、不适合什么、输入输出协议是什么、预算成本是什么。

它只提供选择信息，不产生派发意图。

### Assignment

Lead 派发时必须给出的标准派工单。

字段包括目标、范围、约束、期望输出、预算和回流条件。

### Execution

机器层创建的执行事实记录。

它记录谁派的、谁执行、什么状态、什么时候开始结束、关联哪些 artifact。

### Progress

执行过程中的事实事件。

它可以记录进度、artifact、warning、blocked、heartbeat，但不替 Lead 做战略总结。

### Closeout

执行体最终交回 Lead 的标准报告。

字段包括状态、摘要、证据、改动路径、验证、风险和下一步建议。

### WakeSignal

执行完成、失败、暂停、预算耗尽或中止时写入的单文件唤醒信号。

它只负责唤醒 Lead；Lead 必须读取 execution 记录、closeout、artifact 和验证结果后再判断。

## 通用与定制边界

- `src/protocol/` 只放通用协议。
- `src/team/` 只放 teammate 接入逻辑。
- `src/subagent/` 只放 subagent 接入逻辑。
- `src/workflows/` 只放 workflow 接入逻辑。
- 未来定制能力必须注册为 capability，并遵守 Assignment 与 Closeout。

## 禁止事项

- 禁止机器层因为文本复杂而自动派发。
- 禁止机器层因为某能力可用而自动创建执行体。
- 禁止定制能力绕过 AssignmentContract。
- 禁止执行体绕过 CloseoutContract。
- 禁止把 WakeSignal 当成结果真相。
