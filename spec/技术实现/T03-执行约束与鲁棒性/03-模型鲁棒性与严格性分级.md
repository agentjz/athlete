# 模型鲁棒性与严格性分级

## 1. 目标与边界

本模块只解决五件事：

1. 工具参数 strictness 分层（L0/L1/L2）
2. provider recover 预算截断与回主复核
3. managed continuation 切片预算截断与回主复核
4. 委派评估、机器硬约束与 subagent 硬预算停机
5. 委派等待期间 Lead 轮询与自身推进交替（不对外 pause）

不引入额外产品层，不把策略散落进主循环之外的临时胶水。

## 2. strictness 分层实现

### 2.1 模块边界

- `src/tools/toolArgumentStrictness.ts`
  - tier 判定（L0/L1/L2）
  - L0 unknown args 剥离策略
- `src/tools/toolArgumentContract.ts`
  - schema 合约校验（required/type/enum/unknown）
- `src/tools/toolPrepare.ts`
  - parse -> strictness -> contract -> guards 的串联编排
- `src/tools/registry.ts`
  - 使用 prepare 后的规范化 `rawArgs` 执行工具
- `src/tools/toolFinalize.ts` + `src/types.ts`
  - 将 strictness tier 与 unknown 剥离告警写入协议元数据

### 2.2 行为契约

- L0：先剥离 unknown，再校验 required/type/enum。
- L1/L2：不剥离 unknown，按合约阻断。
- 失败统一 `INVALID_TOOL_ARGUMENTS`，并附结构化 `details`（kind/path）。

## 3. provider recover 预算实现

### 3.1 模块边界

- `src/agent/recoveryBudget.ts`
  - 预算解析（默认值 + 限幅）
  - 预算判定（次数 + 时长）
- `src/agent/runTurn.ts`
  - recover 捕获点接入预算判定
  - 超限写入 `pause.provider_recovery_budget_exhausted`
- `src/agent/turn/managed.ts`
  - Lead 命中 provider budget 后回主调度复核（rebound），而非直接终止 managed loop
- `src/agent/runtimeTransition/builders.ts`
  - provider budget structured transition 构建

### 3.2 状态写回

超限后通过 checkpoint 正式写回（统一结构化记录）：

- `reason.code = pause.provider_recovery_budget_exhausted`
- `attemptsUsed/maxAttempts/elapsedMs/maxElapsedMs/lastError`
- phase 进入 `recovery`
- Lead 主路径遇到该记录后，managed loop 回主调度复核并继续；非 Lead 保持 `pause` 收口

## 4. managed slice 预算实现

### 4.1 模块边界

- `src/agent/turn/managedBudget.ts`
  - 切片预算解析与判定（slices + elapsed）
- `src/agent/turn/managed.ts`
  - 每次 yield 后判定预算是否耗尽
  - 超限写入 `pause.managed_slice_budget_exhausted`
  - Lead 命中后回主调度复核并重置预算窗口

### 4.2 状态写回

超限后 checkpoint 写回（统一结构化记录）：

- `reason.code = pause.managed_slice_budget_exhausted`
- `slicesUsed/maxSlices/elapsedMs/maxElapsedMs`
- phase 进入 `continuation`
- Lead 主路径不直接终止 turn，而是回主调度继续下一轮

## 5. 委派评估、机器硬约束与 subagent 预算实现

### 5.1 模块边界

- `src/orchestrator/delegation/types.ts`
  - 委派评估、闸门、档位与预算结构定义
- `src/orchestrator/delegation/evaluator.ts`
  - 必要性评估（结构化 `necessary/score/reasons`）
- `src/orchestrator/delegation/policyGate.ts`
  - 机器闸门放行/拒绝裁决
- `src/orchestrator/delegation/mode.ts`
  - `快/均衡/深度` 档位阈值与预算映射
- `src/subagent/budget.ts`
  - 工具数/模型轮次/墙钟时长预算跟踪与超限判定
- `src/subagent/errors.ts`
  - 结构化预算超限错误
- `src/orchestrator/returnBarrier.ts`
  - 委派后回主复核屏障
- `src/orchestrator/leadLoop.ts`
  - `wait_for_existing_work` 时把 active delegated work 注入回 Lead，让 Lead 准备合流、整理证据和做非冲突检查
- `src/agent/turn/delegatedWaitRhythm.ts`
  - 委派仍未完成时，阻止连续委派状态轮询；Lead 必须先做一项非冲突的自身推进，再查下一批委派状态；当前目标的 task / worktree 收口查询不算委派状态轮询

### 5.2 行为契约

- 关键词只作为 hint，不再单独触发委派。
- 委派必须先经过 evaluator 记录必要性建议，再经过 policy gate 检查机器硬约束；evaluator 分数不作为硬拦截条件。
- 档位影响 evaluator 倾向、并发委派上限与 subagent 预算。
- subagent 超限时写结构化 `subagent_budget_exhausted`（含维度与快照）并强制返回 Lead。
- 每次委派结束后必须回主控复核，才能继续下一轮委派。
- Lead 命中 `wait_for_existing_work` 时不向宿主返回终态 `pause`。
- Lead 可以用一个工具批次查委派状态；如果结果还没回来，下一步必须先做合流准备、已有证据整理或非冲突检查，再允许继续查委派状态。

## 6. `background_terminate` 幂等实现

### 6.1 模块边界

- `src/execution/background.ts`
  - 终态对象提前返回 no-op
- `src/tools/background/backgroundTerminateTool.ts`
  - 返回契约补充 `already_terminal/idempotent`
  - 终态 no-op 不伪造二次 close 协作事实

### 6.2 契约

对 `completed/failed/timed_out/aborted`：

- `ok=true`
- `already_terminal=true`
- `idempotent=true`
- 不改写 `status/finishedAt` 真相

## 7. 过渡与观测兼容

为保证历史 checkpoint/transition 可兼容读取，同步调整：

- `src/agent/runtimeTransition/normalize.ts`
