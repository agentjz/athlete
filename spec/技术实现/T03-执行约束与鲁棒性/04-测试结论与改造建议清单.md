# 测试结论与改造建议清单

## 1. 本轮结论

### 1.1 已完成改造

- strictness 分层已落地：L0 unknown 宽容剥离，L1/L2 阻断。
- provider recover 已接入次数+时长预算，命中会写结构化 transition；Lead 走回主复核，非 Lead 正式 `pause`。
- managed continuation 已接入 slice+时长预算，命中会写结构化 transition；Lead 走回主复核，非 Lead 正式 `pause`。
- 委派等待已改为机器层事件挂起；等待中不再对外 pause，也不再让 Lead 模型连续空转查状态。
- `background_terminate` 对终态对象已实现幂等 no-op。

### 1.2 已验证的关键行为

- L0 只放松 unknown，不放松 required/type/enum。
- provider recover 命中预算后不再无限重试，且 Lead 不再被预算硬停。
- managed continuation 命中预算后不再无限续跑，且 Lead 会回主复核继续决策。
- 委派执行中 Lead 不再“看起来停机”，也不再消耗模型轮次查状态；机器层等 execution 事实变化后继续交给 Lead 合流复核。
- terminate 重入不抛错且不改写终态真相。

## 4. 已知取舍

- provider recover 的集成测试仍存在重试等待时间（由底层 API retry 策略决定）；Lead 走回主复核语义，非 Lead 仍由预算 pause 明确收口。
- managed slice 预算与 `maxContinuationBatches` 保持默认对齐；Lead 采用“预算窗口 + 回主复核”语义。

## 5. 后续建议

1. 将 provider request retry 的等待策略参数化（便于测试环境缩短等待）。
2. 在运行态摘要里增加 budget 命中计数聚合（方便线上调参）。
3. 若需要更细粒度放松策略，可在 L1 引入显式工具白名单配置，但保持默认阻断。
