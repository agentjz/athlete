# 配置系统

## 文档目的

本文定义 Kitty 当前的配置入口、内部拆分、升级边界和宿主共享规则。

## 模块目标与当前状态

配置系统当前只有一个对外入口：`src/config/store.ts`。所有宿主和核心模块都从这里拿 runtime config。

## 正式边界

主要代码边界：

- `src/config/store.ts`
- `src/config/fileStore.ts`
- `src/config/runtime.ts`
- `src/config/env.ts`
- `src/config/schema.ts`

## 真相源与状态归属

当前配置来源包括：

- 项目 `.kitty/config.json`
- 项目 `.kitty/.env`
- 环境变量覆盖

这些来源由 `runtime.ts` 归一后，产出唯一 runtime config。

## 主路径

当前主路径如下：

1. `fileStore.ts` 负责配置文件读写与 schema 升级。
2. `env.ts` 负责 `.env` 装载。
3. `runtime.ts` 合并文件配置、项目 `.env` 与环境变量。
4. `store.ts` 作为唯一对外入口导出配置能力。

模型配置只有一套核心环境变量：

- `KITTY_PROVIDER`
- `KITTY_API_KEY`
- `KITTY_BASE_URL`
- `KITTY_MODEL`
- `KITTY_THINKING`
- `KITTY_REASONING_EFFORT`

运行预算也必须走同一入口，不能藏在宿主或工具私有逻辑里：

- `KITTY_CONTEXT_WINDOW_MESSAGES`
- `KITTY_MAX_CONTEXT_CHARS`
- `KITTY_CONTEXT_SUMMARY_CHARS`
- `KITTY_MAX_OUTPUT_TOKENS`
- `KITTY_YIELD_AFTER_TOOL_STEPS`
- `KITTY_MAX_TOOL_ITERATIONS`
- `KITTY_MAX_CONTINUATION_BATCHES`
- `KITTY_MANAGED_TURN_MAX_SLICES`
- `KITTY_MANAGED_TURN_MAX_ELAPSED_MS`

Lead、teammate、subagent 是运行身份，不是模型凭据或 provider 维度。身份差异只能体现在 prompt、工具面、账本、执行边界和 closeout 规则上。所有 agent loop 调用同一套核心 provider / key / model。

## 模块职责与非职责

### 配置系统负责

- 统一读取和归一化配置
- 维护 schema version
- 向 provider、Telegram、MCP 等模块输出共享 runtime config

### 配置系统不负责

- 在宿主内部重复读取 `.env`
- 维护第二套临时配置入口
- 按运行身份切换模型、key、provider 或 base URL
- 为历史配置保留长期脏兼容

## 失败路径与异常路径

当前明确处理：

- `schemaVersion` 不支持时 fail closed
- 历史配置只做一次性升级
- 宿主、CLI、tool、service 不得各自读 `.env` 形成第二入口

## 测试与验证

当前主要由以下测试保护：

- `tests/config/provider-runtime-config.test.ts`
- `tests/agent/unified-model-config.test.ts`
- `tests/cli-product-engineering.test.ts`

## 当前落地决定

当前配置系统固定为“项目 `.kitty/` 集中状态 + 单一入口 + schema version + runtime 归一化 + 一套核心模型配置 + 一套运行预算配置”。DeepSeek V4 这类百万级上下文模型的现场上下文、摘要压力、输出 token 上限和长任务节奏必须由 `.kitty/.env` 明确配置，再由 `runtime.ts` 统一读入。Lead、teammate、subagent 不拥有独立模型配置。正式项目配置与运行状态不写入 AppData、Roaming、Local 或其他用户级目录。
