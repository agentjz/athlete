# 工具体系规划与收敛

## 文档目的

本文描述 Kitty 当前的工具面、工具注册边界、文件工具职责分工和测试保护方式。

## 模块目标与当前状态

当前工具专项已经完成以下收口：

- 工具名保持稳定
- 不增加平行工具族
- 默认工具集合保持全量可见，排序不等于裁剪
- `find_files` 作为正式 builtin tool 接入
- `read_file / search_files / list_files / edit_file` 已按当前职责增强
- `code_symbols / code_references / code_pattern` 已作为只读代码事实取证工具接入
- `write_file`、`run_shell` 等高风险工具继续受治理与 guard 约束

## 主路径

### 1. 注册与暴露主路径

当前 builtin tool 统一在 `src/capabilities/tools/core/builtinCatalog.ts` 注册，经 `registry -> governance -> runtimeRegistry` 暴露；宿主 extra tool 先进入 `src/host/toolRegistry.ts`，再进入同一条主路径。
参数 contract 统一在 `toolArgumentContract` 执行：schema 不匹配时在 prepare 阶段 fail closed，不再进入 execute。

工具治理 metadata 只用于 registry 收敛、执行边界、结果信号和能力呈现；不得作为普通执行阶段隐藏工具或替 Lead 选择工具的依据。
turn-time presentation ordering 只能重排工具定义，不能改变工具名集合。
closeout 阶段允许极窄的防打转过滤，但该过滤只服务已接近收口的 task/todo 循环控制，不得外溢为普通执行阶段的工具裁剪机制。

### 2. 文件探索主路径

当前推荐顺序是：

- 看目录：`list_files`
- 按路径找目标：`find_files`
- 按内容搜索：`search_files`
- 按代码事实取证：`code_symbols / code_references / code_pattern`
- 读取正文：`read_file`

系统提示和测试已经明确把文件探索从 shell-first 拉回到正式工具面。

`search_files` 当前支持 `files / count / matches` 三种取证视图：`files` 只返回匹配文件和后续读取参数，`count` 只返回每文件匹配数量，`matches` 返回匹配行和有限上下文。默认视图是 `files`，先控制噪声，再由 Lead 显式决定是否读取正文。该工具只提供搜索事实和继续读取入口，不产生读取意图，不替 Lead 选择目标文件。

代码事实工具当前分为三条正式入口：`code_symbols` 返回声明和文件结构事实，`code_references` 返回标识符引用事实，`code_pattern` 返回结构模式匹配事实。三者都是只读工具，只返回行号、文本片段和后续 `read_file` 参数；它们不承担影响范围判断、修改点选择、验证充分性判断或修复策略生成。

### 3. 文件修改主路径

当前职责分工为：

- 创建文件：`write_file`
- 小范围、多点位精修：`edit_file`
- patch 语义修改：`apply_patch`

`edit_file` 当前要求 `expected_identity` 与 `anchor`。`expected_identity` 只证明编辑身份来自同一目标文件；真正的写入许可由当前文件里的目标 anchor 和 `old_string` 现场复核决定。文件其他位置变化不应让目标未变的编辑失败。`edit_file` 必须保留原文件的 UTF-8 BOM / CRLF / UTF-16 envelope。`write_file` 覆盖已有文件会在 prepare 阶段被阻断，已有文件修改必须走 `read_file + edit_file` 或 `apply_patch`。
三个写工具在写后统一返回 `changedPaths`、diff、diagnostics 和 session diff。

### 4. shell 执行主路径

`run_shell` 当前维持单一工具名，但底层执行结果已结构化为：

- 执行状态（`completed/failed/timed_out/stalled/aborted`）
- 退出码、耗时、重试次数
- 输出截断标记
- 长输出落盘路径与预览

长输出当前在执行过程中就受正式预览上界控制，并在超出阈值后持续落盘，不再依赖命令结束后的事后截断。

`run_shell / background_run / background_check / background_terminate` 共享 `kitty.exec` process contract，统一表达 start/read/terminate/exited/closed 的等价语义。

## 模块职责与非职责

### 工具体系负责

- 暴露正式工具面
- 管理 registry 与 governance
- 保持文件探索与修改工具的分工
- 提供代码声明、引用和结构模式的只读事实
- 让宿主扩展和 MCP 能力进入统一主路径
- 保证默认工具集合不因呈现顺序、workflow hint 或阶段状态被隐式裁剪
- 保证工具呈现只降低噪声，不把技能、浏览器、后台或队友通道变成机器代选

### 工具体系不负责

- 再次定义机器派发规则
- 让宿主各自维护独立工具池
- 通过改名制造第二套能力入口
- 替 Lead 做工具选择或任务策略选择
- 把阶段状态实现为工具可见性开关

## 失败路径与异常路径

当前明确处理：

- 同名工具注册在 registry 边界显式失败
- 缺 governance 的扩展 fail closed
- `write_file` 覆盖已有文件在 prepare 阶段被阻断
- `edit_file` 目标 anchor 已变化时 fail closed；整文件 identity 变化只作为事实反馈，不作为阻断理由
- `run_shell` 直接读取文件内容在 prepare 阶段被阻断
- schema 不匹配的工具参数在 prepare 阶段 fail closed，execute 不会被触发
- `openapi_inspect / openapi_lint` 读取本地 JSON 时走统一文本解码，必须接受 UTF-8 BOM 文档
- `session_list / session_search` 是历史索引工具，遇到损坏 session snapshot 时跳过该条并返回 `skippedSessions`；`session_read / session_final_output` 读取指定损坏 session 仍然 fail closed
- 工具呈现排序必须保持工具集合不变
- 网页呈现排序必须保持轻量 HTTP / download 路线优先于浏览器自动化，浏览器只作为可见能力和交互后备
- closeout 防打转过滤必须被限制在收口路径，普通执行阶段不得受影响
- `run_shell` 长输出会在执行过程中自动控上界并写入状态目录，避免上下文爆炸和单次命令输出失控占用内存
- 不再为 `read / write / edit / bash / grep / ls` 之类平行入口保留位置

## 当前落地决定

当前工具体系的正式决定如下：

- `find_files` 是唯一路径模式查找工具。
- `edit_file` 是结构化精修工具，`apply_patch` 继续保留 patch 语义。
- `read_file` 负责 identity、anchors、文本 envelope 归一化和 continuation 信息。
- `search_files` 负责结构化内容搜索和低噪声匹配分布，不承担完整文件读取。
- `code_symbols / code_references / code_pattern` 负责工程级代码事实取证，不承担策略判断。
- `run_shell` 继续是单一入口，但执行 runtime 与输出收口已正式化。
- `background_terminate` 成为正式 terminate surface，并沿统一 execution closeout 收口。
- 工具体系继续沿 registry、governance 和 host 注入边界增长，不靠平行命名扩张。
- 工具呈现顺序、workflow hint 与治理 metadata 不承担隐藏默认工具集合的职责。
- `load_skill`、浏览器工具、后台工具和团队工具都不承担“机器替 Lead 选路线”的职责；它们只提供能力、信号和证据。
- closeout 过滤只用于防止已接近收口时 task/todo 状态工具反复打转。
