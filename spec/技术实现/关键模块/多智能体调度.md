# 多智能体调度

## 当前定位

Athlete 当前已经不是“有子 Agent 功能”的系统，而是已经进入“总指挥调度”阶段。

## 当前执行角色

- `lead`
- `subagent`
- `teammate`
- `background`

## 当前关键语义

- split
- dispatch
- wait
- merge

## 当前机器化落点

- split 通过 `src/orchestrator/taskPlanning.ts` 写入持久任务图
- dispatch 通过 `src/orchestrator/dispatch.ts` 先改控制面，再决定是否进入下一轮 orchestration
- wait 通过 `pause.orchestrator_waiting_for_delegated_work` 落成正式终态，不再继续偷偷跑 lead turn
- merge 通过 `src/orchestrator/leadInput.ts` 变成 lead 的显式执行阶段

## 当前主路径

`lead` 现在不是“prepare 一下再自己继续干”，而是：

1. `src/orchestrator/leadLoop.ts` 先循环跑 orchestration
2. 只要当前状态还属于 dispatch / wait，就不进入普通 lead turn
3. 只有 route 给出 `self_execute`，lead 才会收到正式阶段输入并继续执行
4. delegated work 完成后，系统必须通过 merge 阶段正式回到 lead

## 当前要求

1. lead 是总指挥，不是永远自己亲自做。
2. 拆分和派工必须落在可恢复的任务图上。
3. wait 必须有机器状态，不靠 prompt 暗示。
4. merge 必须有正式合流点，不允许“做完了但没并回主线”。
5. teammate / background 的 Windows 子进程不应再通过可见 CMD 窗口表达“我被派出去了”；派工状态应由控制面和正式终态表达。
