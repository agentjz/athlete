# 宪法原则

这一组文档是 Athlete 当前唯一有效的架构原则集合。

它们直接服务于实现、重构、删旧和长期维护。

## 当前原则体系

### 核心 12 条

- `P01-一个循环一个智能体.md`
- `P02-加一个工具只加一个处理器.md`
- `P03-先计划再动手.md`
- `P04-大任务拆给子智能体.md`
- `P05-知识按需加载.md`
- `P06-上下文要能压缩.md`
- `P07-任务图要落盘.md`
- `P08-慢操作放后台.md`
- `P09-任务太大就分给队友.md`
- `P10-队友之间要有统一协议.md`
- `P11-队友自己认领任务.md`
- `P12-工作区和任务要隔离.md`

### 当前工程铁律

- `P13-session是任务现场.md`
- `P14-执行约束不是安全策略.md`
- `P15-provider必须可替换.md`
- `P16-配置只能有一个入口.md`
- `P17-扩展靠事件生长.md`
- `P18-主循环和文件都不能长胖.md`
- `P19-先写失败测试再写实现.md`
- `P20-外部事实必须绑定证据.md`
- `P21-没验过就不能收口.md`
- `P22-阶段推进必须有机器状态.md`
- `P23-文本链路必须稳定可读.md`
- `P24-错误兼容不能高于正确性.md`

## 使用规则

1. 新原则必须长期有效，且跨多个模块成立。
2. 临时实现细节不要升格成原则。
3. 若与当前实现冲突，应优先修正文档或实现，不能长期并存。
4. 若只是模块约束，优先写到 `modules/`、`implementation/` 或 `adr/`。
5. Athlete 仍处于高频重构阶段；错误旧逻辑、错误旧测试、错误旧兼容如果阻碍主干修复，应删除或重写，不为错误行为保活。
6. 本项目当前优先级是“正确、稳定、可维护”，不是“尽量少改旧代码”。
7. 若任务存在显式 acceptance / closeout 契约，该契约也是机器真相源的一部分；缺文件、缺证据、缺验证都不能收口。
