# 开发规则

## 总则

Athlete 是 harness 工程，不是一次性脚本。

所有改动优先保护：

1. 耐跑底盘
2. 控制面真相
3. 扩展边界

## 文件规则

1. 单文件默认不超过 300 行。
2. 超过 300 行时必须检查是不是耦合过多。
3. 能拆目录就拆目录，能拆文件就拆文件。
4. 一个文件只做一件主要事情。
5. 一个模块尽量只保留一个主要变更理由。

## 架构规则

1. 控制面判断不偷塞进工具细节。
2. 工具只通过注册表接入。
3. 新状态优先进入现有真相源，不先新建平行 JSON。
4. skills 负责知识与流程，不替代工具动作。
5. 总指挥层负责调度，不负责具体文件改写。

## 重构清理规则

1. Athlete 当前按新项目对待，以当前最优主干为准，不为错误旧兼容保活。
2. 已确认错误、残余、过时、阻碍主干的旧结构，需要删就删，需要清就清。
3. 不为了“平滑一点”长期保留双写、平行状态、过渡层、影子路径。
4. 可以做一次性清理或一次性迁移，但不要引入长期历史包袱。
5. 如果旧测试保护的是错误行为，应删除或重写，不让错误测试绑架主干。

## 文档规则

1. SPEC 写当前真实能力与未来方向，但两者必须分开。
2. docs 偏解释和维护，spec 偏约束和边界。
3. ADR 只记录关键决定，不拿来写宣传。

## 测试规则

1. 关键骨架改动先补失败测试。
2. 修过的真实 bug 尽量固化成回归测试。
3. 不为了覆盖率堆低价值测试。
