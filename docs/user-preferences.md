# Kitty 用户偏好

这个文件只记录项目所有者不要什么。

不要把它写成教程、计划、流程或技术说明书。以后只补充“不要做什么”。

## 最高原则

不要打补丁。

不要糊弄过去。

不要做假兼容。

不要做优雅过渡。

不要在旧东西上缝缝补补。

正确原则是：直接物理删除，再新建。

因为物理删除之后，假兼容、优雅过渡、补丁残留、旧口子这些问题会直接消失。需要的能力再按当前真相重新建立。

## 沟通不要什么

- 不要用非简体中文沟通。
- 不要长篇大论。
- 不要绕弯子。
- 不要说空话。
- 不要编造事实。
- 不要把猜测说成事实。
- 不要在没有验证时说完成。
- 不要向项目所有者倾倒代码细节。
- 不要用术语包装简单事实。
- 不要把内部实现当成回答。
- 不要粘贴大段日志、diff 或源码，除非项目所有者明确要求。

## 做事不要什么

- 不要迎合用户一句话就乱改。
- 不要机械执行用户字面指令。
- 不要只盯一个局部就动手。
- 不要没有 research 就判断。
- 不要为了删而删。
- 不要为了减而减。
- 不要为了集中而集中。
- 不要软绵绵地一点点改。
- 不要做优雅过渡。
- 不要做假兼容。
- 不要打补丁糊过去。
- 不要保留旧口子。
- 不要把旧东西换个名字留下来。
- 不要用脚本批量替换源码。

## 架构不要什么

- 不要把 Agent 当小孩。
- 不要把系统做成保姆。
- 不要给模型上太多限制。
- 不要让 runtime 替模型规划路线。
- 不要让记录层变成第二个脑子。
- 不要把扩展能力塞进默认核心。
- 不要让旧设计以 route、prompt、flag、mode、test、doc、fallback、wrapper、adapter、alias 的形式残留。

## 取舍不要什么

- 不要为了短期看起来稳定而留下长期复杂。
- 不要为了避免震动而保留错误设计。
- 不要把“能跑”当成“设计对”。
- 不要把“保留以后也许有用”当成理由。
- 不要把“先兼容着”当成默认选择。
- 不要用 V0、V1、MVP、最小产品、最小入口、最小切口当借口交半成品。
- 不要只按眼前一个扩展设计；要按未来可以承载大量扩展的结构设计。
- 不要为了快而做薄弱骨架。
- 不要把“先跑起来”包装成“架构完成”。
- 不要为空的未来生态预留空壳。
- 不要把还不存在的扩展族群写成空目录、空文件、空 registry、空 provider。
- 当前只有什么，就只写什么；未来新增时再物理新建。
