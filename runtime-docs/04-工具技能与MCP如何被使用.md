# 工具、技能与 MCP 如何被使用

## 一句话

工具是手，技能是方法，MCP 是外部设备。  
它们都给 Lead 使用，但都不替 Lead 指挥任务。

Deadmouse 不靠把所有规则塞进 prompt 里变强。  
它靠清楚的能力入口变强。

## 为什么需要工具

如果没有工具，模型只能“说自己做了什么”。

用户让它看文件，它只能猜。  
用户让它跑测试，它只能描述应该怎么跑。  
用户让它改代码，它只能写一段建议。  
用户让它查网页，它只能根据记忆回答。

这就很容易变成假动作：嘴上说完成，实际上没有碰到真实世界。

工具的作用是把话变成动作。  
读文件是真的读。  
改文件是真的改。  
跑命令是真的跑。  
查结果是真的拿结果回来。

工具返回以后，Lead 再看结果，决定下一步。

## 运行故事：读文档

用户说：“帮我看这个 docx 文档，总结问题。”

Lead 不应该一上来就被机器拦住，说“必须先加载某某技能”。

更好的运行方式是：

Lead 先确认文件在哪里。  
确认它是不是 docx。  
确认能不能读取。  
然后再决定要不要加载文档阅读技能。  
最后用合适的工具读取内容。

skill 像方法。  
它告诉 Lead：读这种文档要注意标题、结构、表格、批注、结论和风险。

tool 像手。  
它真正去读文件、跑命令、改文件、查结果。

如果没有 skill，Lead 也可以用工具做事，但可能少一套专门方法。  
如果只有 skill 没有 tool，Lead 知道方法，却不能真正读取文件。  
两者合起来才像一套完整工作方式：skill 告诉它怎么做，tool 帮它真的做。

## 运行故事：查网页

用户说：“查一下这个网页。”

Lead 可以先用轻量方式抓网页内容。  
如果页面很简单，不一定需要浏览器。

如果页面要点击、登录、等待加载，再使用浏览器能力。

这里的重点不是“网页就必须上浏览器”。  
重点是 Lead 根据现场选工具。

## 运行故事：接外部能力

有些能力不是系统自带的，比如浏览器自动化、外部服务、特殊工具。

这些可以通过 MCP 接进来。

MCP 像外部设备。  
需要时接上，用完回到主线。  
它不应该让主循环变胖，也不应该绕开 Lead 自己做决定。

## 扩展为什么不能都塞进 prompt

如果每加一个工具、一个技能、一个宿主入口，都往提示词里加一段规则，系统会越来越胖。

最后会变成：主循环越来越长，宿主偷偷拼逻辑，新能力只能靠胶水硬塞。

Deadmouse 的思路是：新能力长在正式入口上。  
工具走工具入口，技能走技能入口，MCP 走外部能力入口，宿主走宿主入口。

## 这里的重点

能力层不是指挥层。

它只负责让 Lead 有更多手段。  
Lead 仍然要看目标、看现场、看证据，再决定用什么能力。
